# AionCore 5.8 事件响应流程 / Incident Response

> **目标**：发生故障时人不慌、流程不乱、复盘有产出。  
> **配套**：[`./runbook.md`](./runbook.md)（具体处置）· [`./architecture.md`](./architecture.md) · [`./observability.md`](./observability.md)  
> **最高原则**：**Blameless** — 事后只问"系统/流程哪里漏了"，不问"谁错了"。

---

## 1. 事件分级 / Severity

| 级别 | 标准 | Ack 时限 | Resolve 目标 | 通报范围 |
|------|------|---------|-------------|----------|
| **P0** | 全服 down / 数据丢失 / 玩家资产丢失 / 安全入侵 | **15 min** | **1 h** | 全员 + 玩家公告 |
| **P1** | 单 service 异常 / 部分功能不可用（副本/邮件/拍卖等单系统） | 1 h | 4 h | 内部 + 视情况公告 |
| **P2** | 性能下降 / 个别玩家报障可复现 | 8 h | 24 h | 内部 |
| **P3** | 体验问题 / 数据轻微不一致 / 文档错漏 | 72 h | 下个迭代 | issue tracker |

### 升级触发（自动升一级）
- P1 持续超过 resolve 目标 → 升 P0
- P2 影响人数 > 10% 在线玩家 → 升 P1
- 任何级别出现"数据不可恢复"迹象 → 升 P0

### 降级条件
- 必须等到「根因明确 + 临时缓解生效 + 监控指标回归」三项都满足才能降级
- 仅"看起来正常了"**不构成**降级理由

## 2. 响应流程 / Response Flow

```
检测 (alert / 玩家反馈)
    │
    ▼
[Ack within SLA] ── 内部通报"我接手了"
    │
    ▼
[Triage] ── 定级 P0/1/2/3 → 跳到 runbook 对应 entry
    │
    ▼
[Mitigate] ── 先止血（重启 / 回滚 / 切流量）
    │
    ▼
[Communicate] ── 玩家公告（P0/P1）+ 内部进展更新
    │
    ▼
[Root cause] ── 不要急着关单；找到根因再说
    │
    ▼
[Resolve + 验证] ── 监控指标 + 玩家抽样验证
    │
    ▼
[Postmortem] ── 24h 内出文档（P0/P1 强制）
```

**铁律**：
- ❌ **不要在 root cause 没找到时就关单** — 否则一定复发
- ❌ **不要在玩家不知情时静默修复** — 玩家发现"我刚才进不去"会怀疑外挂/封号/被骗
- ✅ **先止血再追因** — 玩家时间比你的好奇心贵

## 3. 沟通模板 / Communication Templates

### 3.1 内部通报（QQ 群 / 钉钉 / 飞书）

**初始通报**（Ack 时发）：
```
[P{级别}] {一句话症状}
影响: 估计 {N} 玩家 / {时间起点}
处置: {你的名字} 接手中，目标 {时限} 内 ack 根因
临时缓解: {正在做什么}
更新频率: 每 {15/30/60} 分钟
```

**进展通报**（每周期发一次，直至 resolve）：
```
[P{级别}] 进展更新 ({HH:MM})
状态: 已止血 / 仍在排查 / 等待 {依赖}
新发现: {本周期新信息}
下一步: {接下来 15/30 min 做什么}
```

**Resolve 通报**：
```
[P{级别}] 已解决 ({HH:MM})
影响时长: {分钟}
影响玩家: {人数 / 估计}
根因: {一句话}
已落地: {临时修复}
后续: postmortem 24h 内出，action items 跟进
```

### 3.2 玩家公告（开服公告 / 群公告 / launcher）

**P0 紧急维护**：
```
【紧急维护通知】
我们于 {HH:MM} 检测到服务器 {简述故障}，已暂停服务以保护您的角色数据。
预计恢复 {HH:MM}（以本公告更新为准）。
恢复后将提供 {补偿方案}。
服务期间您的角色数据完全安全，请耐心等待。 —— 拾光AI 运营组
```

**P1 部分功能不可用**：
```
【系统提示】
{副本 / 邮件 / 拍卖 / 其他} 系统当前异常，正在处理。
其他功能（{枚举}）不受影响。预计 {HH:MM} 前恢复。 —— 拾光AI 运营组
```

**Resolve 通知**：
```
【恢复公告】
{HH:MM} 起 {故障范围} 已恢复。
原因：{对玩家友好的解释}  影响时长：{分钟}
补偿：{具体补偿}（{HH:MM} 起游戏内邮件领取）
感谢理解。 —— 拾光AI 运营组
```

### 3.3 状态页（未来引入时）

字段：组件（gateway/world/chat/PG/Redis/NATS）× 状态（Operational / Degraded / Outage / Maintenance）× 最近更新。

## 4. Postmortem 模板（Blameless）

> P0/P1 强制，24h 内首版；P2/P3 可选。  
> 文件位置：`server/doc/postmortems/YYYYMMDD-{slug}.md`

```markdown
# Postmortem: {一句话事件标题}

- 事件 ID / 级别 / 发生时间 / 影响时长 / 影响范围 / 作者 / 审阅 / 状态

## 摘要 / TL;DR
{3 句话：什么坏了 → 多大影响 → 怎么修的}

## 时间线 / Timeline (精确到分钟)
| 时间 | 事件 |（监控告警→ack→定位组件→临时缓解→玩家影响停止→根因→永久修复→关单）

## 影响 / Impact
玩家数 / 持续时长 / 资产损失（是否补偿） / 数据完整性 / 声誉

## 根因 / Root Cause
{完整技术因果链：直接原因 → 触发条件 → 系统设计漏洞}

## 解决步骤 / What We Did
1. {第一步动作 + 时间 + runbook entry / 临场决策}

## 关键决策点 / Key Decisions
为什么选 X 不选 Y？事后看正确吗？

## 教训 / Lessons Learned
做对的 / 做错的 / 侥幸的

## Action Items
| ID | 事项 | Owner | 截止 | 状态 |
> 必须**可验收**（"提高代码质量"不算 — "为 module X 添加 race detector 并 CI gate"才算）。

## 如果重来一次 / Hindsight
{2-3 段：发现速度？决策准度？流程顺度？}

## 链接 / References
Grafana 截图 / 相关 commit / 相关 issue / runbook entry / 上次类似事件
```

## 5. 已发生事件参考 / Past Incidents

### INC-20260411-01 — PostgreSQL 勒索（P0，跨工作区）
- **时间**: 2026-04-11
- **范围**: 跨整个 拾光AI 工作区（不限于 ACE_5.8）
- **症状**: PG 数据库被加密 / 留勒索 README，玩家数据不可读
- **根因**: PG 监听 `0.0.0.0:5432`，弱密码（或公网可达）被自动化扫到入侵
- **缓解**: 从冷备恢复（部分数据丢失）
- **永久修复**:
  - PG 永远 `listen_addresses = '127.0.0.1'`（已落地为 CLAUDE.md 全局铁律 #6）
  - `pg_hba.conf` 仅允许 127.0.0.1
  - 新增冷备策略（每日 pg_dump → 异机存储）
- **教训**: "默认配置 + 公网部署"是致命组合。任何数据库默认按"内网 only"配，需要外部访问时必须用 SSH tunnel 或 VPN
- **当前状态**: ✅ 防御已落地；红线写入 [`../CLAUDE.md`](../CLAUDE.md) 关键约束 #2

### INC 模板（其他历史事件待补）
- 当首次有新事件落地时，把 postmortem 链接 + 一句话摘要插到这里
- 保持时间倒序

## 6. 演练 / Drill 建议

> 不演练的应急响应 = 没有应急响应。每次演练后更新 runbook + 本流程文档。

### 月度（每月 1 次，~30 min）
- **PG 重启演练**：在 dev 环境 `net stop postgresql-x64-16` → 观察 world/gateway 行为 → 走 [E3](./runbook.md#e3--postgresql-不可达) 恢复
- **NATS 重启演练**：`taskkill //F //IM nats-server.exe` → 观察 lag → `make boot` 恢复
- **指标检查**：手工拉一次 `curl :9090/metrics`，验证关键指标都在

### 季度（每季 1 次，~2 h）
- **故障注入**：
  - `pkill -9 world.exe` 在玩家高峰模拟（dev 用 tinyclient 模拟玩家）
  - `iptables -A INPUT -p tcp --dport 5432 -j DROP` 模拟 PG 网络分区
  - Redis FLUSHDB 模拟 session 全失效
- **走完整 postmortem 模板**（演练事件）

### 年度（每年 1 次，~半天）
- **完整灾难恢复**：从冷备 / git 清空 dev 机器 → 重新部署 → 玩家可登录
- **人员更替演练**：让从未做过应急的人按 runbook 处置一个注入故障

### 演练记录
- 文件位置：`server/doc/drills/YYYYMMDD-{slug}.md`
- 必含：场景 + 参与人 + 实际处置时间 vs 目标 + runbook 缺漏

## 7. On-Call 值班约定（团队规模 > 1 人时启用）

- **当前**：单人项目，作者本人即 on-call
- **未来**（≥ 2 人时）：
  - 一周轮换
  - 主备双人，主响应、备 backup
  - 交接清单：在跑的告警 / 进行中的事件 / 已知风险窗
  - 5 分钟内 ack 即视为响应；超时自动 page 备班

## 8. 与 Runbook 的边界 / Boundary

| 这个文档 ([incident-response.md](./incident-response.md)) | [runbook.md](./runbook.md) |
|----------------------------------------------------------|----------------------------|
| **流程层面**：分级、沟通、复盘、演练 | **操作层面**：症状 → 命令 → 处置步骤 |
| 回答 "怎么协调一次故障响应" | 回答 "gateway 不响应时跑哪条命令" |
| 不变（除非组织流程变） | 频繁更新（每次新故障 entry） |
